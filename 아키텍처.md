# GlowLink 아키텍처 - 화장품 용기 제조 영업부 통합 자동화 SaaS

## 1. 시스템 개요 및 목적
화장품 용기 제조업체의 영업부(부서장, 영업, 샘플, 지원)의 업무 흐름을 통합하고, 데이터 기반 자동화된 보고 및 실시간 협업을 제공하는 멀티테넌트 SaaS 플랫폼입니다.

## 2. 기술 스택
- **Frontend**: Next.js 15 (App Router), TypeScript, Vanilla CSS / Tailwind CSS (Premium UI)
- **Backend/DB**: Supabase (PostgreSQL, Auth, Edge Functions, Storage)
- **Deployment**: Vercel (Frontend & Server Actions), GitHub (버전 관리 및 CI/CD)
- **Database Connection**: Supavisor 커넥션 풀러 (Port 6543) 사용

## 3. Vercel 배포 및 GitHub 연동 오류 원인 및 해결 아키텍처 (핵심)
기존에 Vercel 빌드 에러나 GitHub 연동 불가가 발생했던 주된 원인은 **Next.js의 빌드 시스템(TypeScript/Webpack)이 Supabase(Deno 기반)의 코드를 함께 컴파일하려고 시도했기 때문**이며, 프로젝트 Root 설정이 올바르지 않으면 발생합니다.

이를 해결하기 위해 다음의 3가지 원칙을 아키텍처에 강제합니다:
1. **`.vercelignore` 분리**: Vercel이 Next.js 빌드를 진행할 때 `supabase/` 폴더를 완전히 무시하도록 `.vercelignore` 파일에 `supabase/`를 명시합니다.
2. **`tsconfig.json` 충돌 방지**: Next.js의 타입스크립트 컴파일러가 Deno 문법을 사용하는 Edge Functions에서 에러를 뿜지 않도록 `"exclude": ["supabase", "node_modules"]` 항목을 추가하여 격리합니다.
3. **단일 루트(Root) 구조**: 프로젝트 최상위 폴더(`GlowLink/`)에 `package.json`, `next.config.mjs`가 위치하도록 하여, Vercel이 GitHub 레포지토리를 가져올 때 즉시 Next.js 프레임워크로 감지하고 정상 배포(Build Command: `next build`)를 진행하게 합니다.

## 5. 데이터베이스 및 RLS 보안 (JOIN-less Strategy)
모든 RLS 정책은 성능을 위해 타 테이블을 JOIN하지 않으며, JWT의 `app_metadata`를 사용해 접근을 제어합니다.
- **인덱싱**: 모든 테이블의 `tenant_id` 및 사용자 식별 컬럼에는 B-tree 인덱스 생성 필수.

### 5.1 핵심 테이블 (Domain Schema)
1. `tenants`: 조직 정보
2. `profiles`: 유저 역할 및 접속 권한 (DEPT_HEAD, SALES, SAMPLE, SUPPORT)
3. `sales_orders`: 수주 내역 및 납기 관리
4. `sample_requests`: 샘플 제작 프로세스 워크플로우
5. `financial_records`: 매출, 원가, 마감 데이터
6. `automated_reports`: Edge Function 등을 통해 자동 생성된 리포트 (JSONB 저장)

### 5.2 RLS 정책 최적화 예시
```sql
CREATE POLICY "Tenant-based access" ON public.[table_name]
FOR ALL TO authenticated
USING (
  tenant_id = (SELECT ((auth.jwt() -> 'app_metadata') ->> 'tenant_id')::UUID)
)
WITH CHECK (
  tenant_id = (SELECT ((auth.jwt() -> 'app_metadata') ->> 'tenant_id')::UUID)
);
```

## 6. 개발 및 보안 자동화 전략
- **Service Role Key 규칙**: 절대로 클라이언트 사이드 코드(React Components)에 노출하지 않으며, 인증이 우회되는 이 키는 Server Actions 또는 Edge Function 내부에서만 제한적으로 사용합니다.
- **Vercel Environment Variables**: 로컬의 `.env.local` 내용물은 전부 Vercel 대시보드 Environment Variables에 빠짐없이 매핑해야 합니다 (URL, ANON_KEY 등).
- **모듈화**: 하나의 파일(`page.tsx` 등)에 거대한 코드를 작성하지 않고 역할을 철저히 분산시킵니다. Serverless Function의 용량 최적화 및 타임아웃 방지를 위해 목적별 컴포넌트로 나눕니다.
